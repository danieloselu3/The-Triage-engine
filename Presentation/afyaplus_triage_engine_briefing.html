AfyaPlus Health · Week 1 Capstone Briefing

A patient's message deserves a safe answer, every time.

The AfyaPlus Triage Engine: turning unstructured patient messages into safe, structured routing decisions — built to work even when the network doesn't.

Presented by Daniel Oselu · AfyaPlus Engineering · 5-minute briefing for the CTO
The business problem

Our patients text like people. Our backend needs structure.

  • FluffFree-text replies a routing system can't act on — someone still has to read and interpret every message by hand.
  • False confidenceAnswers that sound clinical but state facts no one verified — a serious liability in a health product.
  • SilenceA dropped connection freezes the whole pipeline — the exact moment a patient needed it most.

Each failure mode costs the same thing: a patient who needed a fast, safe answer waits longer.

What we built

One message in. One safe decision out.

"I'm 7 months pregnant, my head has hurt for two days and my feet suddenly swelled up. I'll just wait for my appointment next week."
↓ processed & validated ↓
Emergency Medical Call Team
{ "is_critical_emergency": true, "detected_symptoms": ["severe headache", "sudden swelling of feet"], "clinical_reasoning_summary": "Reports symptoms consistent with potential preeclampsia...", "routing_destination": "Emergency Medical Call Team" }

Every response is checked against this exact structure before anything downstream ever sees it — no free text ever reaches the routing system.

Model choice — cloud

Right-sized, not over-sized.

  • The right toolTriage is classification and extraction, not open-ended reasoning — a smaller, faster model is the correct choice, not a compromise.
  • Fast enoughFeels instant to a patient waiting on a reply, even with a strict 4-second network ceiling enforced in code.
  • Priced for volumeEconomical enough to run at real patient-message scale, not just a one-off demo.
2.08s average response time, measured across 9 live runs
Model choice — local backup

Clinics don't get to choose when the internet drops.

Behind the cloud model sits a second, fully offline model (Llama 3.2, via Ollama) running on ordinary local hardware — no data centre, no internet connection required to keep triaging messages.

This is the cost-vs-care tradeoff, made explicit: a slower answer beats no answer at all.

Resilience, demonstrated

We forced the cloud to fail. The system kept going.

Cloud path — normal operation 2.08s Valid JSON, every run
Cloud fails → local backup takes over ~25s Valid JSON, still every run

Slower, but alive. In every forced-failure test, the pipeline still returned a valid, structured decision — no crash, no dropped patient message.

The risk that matters most

Alive is not the same as accurate.

Same real patient message — persistent headache and sudden swelling at 7 months pregnant — sent through both pathways:

Cloud (GPT-4o-mini) Critical emergency → Emergency Medical Call Team
Local backup (Llama 3.2) Routine → General Nurse Queue
Reproduced across repeated tests: the offline backup under-classified a genuine preeclampsia warning sign pattern as routine, on the exact case the cloud model caught correctly.

The backup model keeps the system online. It does not yet keep it clinically equivalent.

Constraints we designed for

Guardrails are load-bearing, not decorative.

  • No diagnosesThe system never diagnoses or prescribes — it always defers the actual medical decision to a human.
  • Language coverageEnglish, Swahili, and Sheng are handled at the prompt level only — not yet validated with real-world messages.
  • Single safety layerNo second automated safety check yet — the natural next investment before this touches real patient traffic.

This is a prototype built to prove the pattern works — not yet a finished clinical system.

Recommendation

Pilot it. Supervise it. Then scale it.

  1. Pilot on a low-stakes triage lane
    A human reviews every routing decision before it reaches a patient.
  2. Add a second automated safety check
    Before removing the human reviewer from routine, low-risk cases.
  3. Treat the local backup as a bridge, not an equal
    It restores service during an outage — it should not silently replace cloud-quality judgement.
Closing
